# Whisper Caravan v0.6 — Structured RAG NPC Reaction Layer

## Status

In progress.

This version builds on the accepted `v0.5-beta-route-pool` baseline.

Slice 1 is complete in code: shared structured reaction types, deterministic local fallback reactions, validation/clamping helpers, and pure reaction application logic now exist without requiring a backend at runtime.

Slices 2 through 4 are complete in code:

* Slice 2 added a backend structured reaction endpoint with deterministic fake responses.
* Slice 3 connected that endpoint to the existing session-based retrieval stack when session-backed memories are available.
* Slice 4 added an optional dialogue/tone provider boundary that can only affect presentation fields and falls back deterministically when missing or invalid.

Slice 5 is complete in code:

* Relevant event cards now surface one deterministic suggested v0.6 NPC when a strong match exists.
* The UI now calls `getNpcReaction()` with the current run `sessionId`, known memory ids, route hint, and compact game state summary.
* Valid structured reactions are applied through `applyNpcReaction()`.
* Collected structured evidence now appears in a lightweight notebook plus a non-authoritative trial preview.

v0.5 established the authored 14-day structure, route-dependent event pool, deterministic scene plan, Bear Court trial, trial scoring, endings, and frontend-only playability.

v0.6 begins replacing frontend mock reaction logic with a backend-driven structured RAG layer.

The goal is not to let AI generate the whole story. The goal is to let RAG retrieve relevant memories/evidence and return structured NPC reactions that can affect gameplay in controlled, deterministic ways.

---

## Core Goal

v0.6 introduces a structured NPC reaction system.

RAG must not only return dialogue text.

Each NPC reaction should return gameplay-relevant structured data, such as:

```json
{
  "npc_id": "fox_ledger_master",
  "dialogue": "I do not remember you, but this ledger says you paid for the medicine.",
  "tone": "guarded",
  "trust_delta": 8,
  "price_modifier": 0.9,
  "quest_available": true,
  "legal_risk_delta": -5,
  "route_unlocks": ["merchant", "truth"],
  "evidence": [
    {
      "memory_id": "ledger_003",
      "title": "Fox Ledger",
      "type": "record",
      "reliability": 0.82,
      "relevance": 0.91
    }
  ],
  "memory_refs": ["ledger_003"],
  "flags_set": ["fox_ledger_seen"]
}
```

This reaction data may affect:

* NPC dialogue tone
* NPC trust
* shop prices
* quest availability
* route unlocks
* legal risk
* evidence notebook
* final Bear Court trial scoring indirectly through collected evidence

---

## Design Rule

The LLM must not directly mutate `GameState`.

The backend or RAG layer may propose structured reaction values, but all gameplay effects must be validated, clamped, and applied by deterministic domain logic.

Correct flow:

```txt
player input
→ reaction request
→ retrieval
→ NPC access filtering
→ structured NPC reaction
→ schema validation
→ deterministic clamping
→ applyNpcReaction()
→ GameState update
```

Incorrect flow:

```txt
LLM output
→ direct GameState mutation
```

## New Shared Types

v0.6 should add or refine shared TypeScript types.

Suggested types:

```ts
export type NPCReactionTone =
  | "friendly"
  | "guarded"
  | "fearful"
  | "hostile"
  | "sympathetic"
  | "evasive";

export type RouteUnlock =
  | "truth"
  | "merchant"
  | "rumor"
  | "failure";

export type EvidenceSummary = {
  memory_id: string;
  title: string;
  type:
    | "record"
    | "testimony"
    | "rumor"
    | "contradiction"
    | "receipt"
    | "law"
    | "inventory"
    | "medical";
  reliability: number;
  relevance: number;
};

export type ReactionRequest = {
  npc_id: string;
  player_input: string;
  day: number;
  route?: string;
  known_memory_ids: string[];
  game_state_summary: {
    trust?: Record<string, number>;
    legal_risk?: number;
    silver?: number;
    flags?: string[];
    unlocked_routes?: string[];
  };
};

export type NPCReaction = {
  npc_id: string;
  dialogue: string;
  tone: NPCReactionTone;

  trust_delta: number;
  legal_risk_delta: number;

  price_modifier?: number;
  quest_available?: boolean;
  route_unlocks?: RouteUnlock[];

  evidence: EvidenceSummary[];
  memory_refs: string[];

  flags_set?: string[];

  explanation?: {
    public_reason: string;
    debug_reason?: string;
  };
};
```

## Deterministic Clamping Rules

All structured reaction values must be validated before being applied.

Recommended safe ranges:

```txt
trust_delta:       -10 to +10
legal_risk_delta:  -8 to +8
price_modifier:   0.75 to 1.25
reliability:       0 to 1
relevance:         0 to 1
```

Invalid route ids must be ignored.

Invalid evidence entries must be discarded.

Missing optional fields should not crash the game.

  request: ReactionRequest

* If backend is absent, use deterministic local fallback.
* Existing frontend-only playability must remain intact.
* Existing `mockData.ts` should not be removed in this version.

## GameState Application Goal

Add deterministic reaction application logic.

Suggested function:

```ts
applyNpcReaction(gameState, reaction)
```

It should update:

* NPC trust
* legal risk
* evidence notebook
* route unlock flags
* price modifiers
* quest availability flags
* custom flags from `flags_set`

This function must be testable and deterministic.

It should not call the backend or LLM.

## Retrieval Design

RAG should retrieve candidate memories, but deterministic rules should decide which memories the NPC can access.

Each NPC should have access boundaries.

Example:

```ts
const npcAccessRules = {
  fox_ledger_master: ["fox_market", "ledger", "receipt", "merchant"],
  crow_broker: ["rumor", "road", "public", "contradiction"],
  camp_healer: ["refugee", "medical", "necessity", "testimony"],
  village_apothecary: ["deer_village", "medicine", "inventory", "truth"]
};
```

An NPC should not reveal memories outside their believable knowledge scope.

---

## Trial Integration Principle

RAG should not directly decide the ending.

RAG affects the trial indirectly by changing what evidence the player has collected.

The Bear Court trial remains deterministic.

Suggested scoring influence:

```txt
trial_score =
  truth_evidence_score
  + testimony_score
  + ledger_score
  + medicine_record_score
  + necessity_score
  - contradiction_penalty
  - legal_risk_penalty
  + npc_trust_bonus
```

Collected evidence from NPC reactions can feed into these categories.

The final ending should still be resolved by `trialLogic.ts`.

## Non-Goals

v0.6 should not:

* Replace the 26-card event pool.
* Let AI generate the full 14-day story.
* Let LLM directly mutate `GameState`.
* Remove frontend-only fallback.
* Rewrite the Bear Court ending system.
* Make every NPC dynamic at once.
* Require backend to play the game locally.

---

## Acceptance Criteria

v0.6 is successful when:

* The game still runs without backend.
* Four first-pass RAG NPCs are represented in shared data/types.
* Frontend can request an `NPCReaction`.
* Local fallback returns valid structured reactions.
* Reaction output is validated and clamped.
* `applyNpcReaction` deterministically updates game state.
* Evidence from reactions can be stored for later trial scoring.
* Backend integration path is clear, even if not fully RAG-powered yet.
* Tests cover clamping, invalid reaction handling, and state application.
