# Whisper Caravan v0.7 — Gemini Dialogue Provider Integration

## Status

Complete in code.

This version builds on the completed `v0.6` baseline.

## Intent

Connect a real LLM provider to the existing structured NPC reaction backend without weakening deterministic gameplay authority.

The goal of `v0.7` is not to let Gemini decide game state.

The goal is to let Gemini improve the presentation layer for structured NPC reactions by generating better:

- dialogue
- tone
- optional public-facing explanation text

All gameplay deltas must remain deterministic and validated locally.

## Scope

- Gemini-backed dialogue provider inside the existing FastAPI backend
- provider selection through environment variables
- structured JSON-only override for presentation fields
- deterministic fallback when Gemini is missing, invalid, slow, or unavailable
- backend tests that prove gameplay fields remain unchanged
- local run instructions for backend-assisted Gemini testing

## Design Rule

Correct flow:

```txt
player input
→ POST /npc/reaction
→ retrieval-backed or deterministic structured reaction
→ Gemini dialogue override attempt
→ schema validation
→ deterministic fallback when invalid
→ structured reaction response
→ applyNpcReaction() in frontend
```

Incorrect flow:

```txt
player input
→ Gemini
→ direct trust / route / evidence / legal risk mutation
```

## Required Behavior

- `POST /npc/reaction` remains the only frontend-facing NPC reaction endpoint.
- The backend must continue to produce a valid structured reaction even when no LLM provider is configured.
- Gemini may override only:
  - `dialogue`
  - `tone`
  - `explanation.public_reason`
- Gemini must not override:
  - `trust_delta`
  - `legal_risk_delta`
  - `price_modifier`
  - `quest_available`
  - `route_unlocks`
  - `evidence`
  - `memory_refs`
  - `flags_set`
- Invalid Gemini output must fall back to the deterministic reaction already produced by the backend.
- Unknown NPC ids must remain safe and non-crashing.
- Frontend-only local play must still work with no backend and no API key.

## Backend Integration Plan

### Slice 1: Provider Boundary Cleanup

- keep `backend/app/dialogue_provider.py` as the only LLM integration boundary
- preserve the current stub provider for tests
- extend provider selection to support:
  - `stub`
  - `gemini`

### Slice 2: Gemini Provider

- add a Gemini-backed implementation in `backend/app/dialogue_provider.py`
- read:
  - `WHISPER_CARAVAN_LLM_PROVIDER`
  - `GEMINI_API_KEY`
  - `WHISPER_CARAVAN_GEMINI_MODEL`
- send compact structured prompt context built from:
  - request metadata
  - retrieved evidence titles/types
  - deterministic fallback dialogue
  - fallback tone
  - deterministic public reason
- require JSON output with only:
  - `dialogue`
  - `tone`
  - `public_reason`

### Slice 3: Validation and Fallback

- reject empty dialogue
- reject invalid tone values
- ignore unknown fields
- return deterministic fallback if:
  - the API key is missing
  - the provider call fails
  - the provider returns invalid JSON
  - the provider returns unsupported tone values

### Slice 4: Tests and Local Verification

- backend unit tests for:
  - no-provider fallback
  - stub-provider override
  - Gemini-provider failure fallback
  - Gemini-provider valid override
  - guarantee that gameplay fields remain deterministic
- manual backend-assisted smoke test through the existing browser NPC interaction flow

## Suggested Environment Variables

```txt
WHISPER_CARAVAN_LLM_PROVIDER=gemini
GEMINI_API_KEY=your_api_key
WHISPER_CARAVAN_GEMINI_MODEL=gemini-3.5-flash
```

## Success Criteria

- The backend can call Gemini through the existing `POST /npc/reaction` flow.
- Gemini-generated dialogue appears only when the backend is enabled and configured.
- Missing or invalid Gemini configuration does not break the endpoint.
- Deterministic gameplay fields remain identical before and after dialogue override.
- Existing frontend structured NPC interactions continue to work without any request-shape changes.

## Delivered in Code

- `backend/app/dialogue_provider.py` remains the only LLM provider boundary.
- Provider selection now supports:
  - `stub`
  - `gemini`
- The Gemini path uses the official Gemini `generateContent` HTTP API with JSON-only response requirements.
- Gemini override attempts are limited to:
  - `dialogue`
  - `tone`
  - `public_reason`
- Backend fallbacks remain deterministic when:
  - no provider is configured
  - `GEMINI_API_KEY` is missing
  - the Gemini request fails
  - Gemini returns invalid JSON
  - Gemini returns an invalid tone
- Backend tests now cover:
  - safe fallback with no provider
  - stub override behavior
  - Gemini missing-key fallback
  - Gemini transport failure fallback
  - valid Gemini override without gameplay-field mutation

## Non-Goals

- No direct LLM ownership of game state
- No free-form chat system
- No frontend-to-Gemini direct calls
- No replacement of retrieval, rules, or trial scoring with model output
- No save/load, auth, or hosted production infra work unless separately scoped
