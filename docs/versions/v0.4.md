# v0.4 Scope

## Intent

Replace deterministic mock retrieval with a real retrieval service while keeping the consequence model interpretable.

## Status

v0.4.0 slice 1 is implemented:

- FastAPI backend skeleton
- `/health`
- `/memories/ingest`
- `/memories/query`
- `/npc/reaction`
- in-memory store only
- metadata filtering only
- frontend retrieval adapter
- local deterministic fallback preserved
- backend-returned evidence metadata shown in UI

v0.4.0 slice 2 is implemented:

- SQLite persistence for ingested memory metadata
- same ingest/query/reaction API contract preserved
- deterministic retrieval rules remain in `backend/app/rules.py`
- local development database path defaults to `backend/data/whisper_caravan.db`

Not started yet in v0.4:

- Chroma
- semantic retrieval
- LLM generation

## Backend Setup

Create a local virtual environment:

```bash
python3 -m venv .venv
```

Activate it:

```bash
source .venv/bin/activate
```

Install backend dependencies:

```bash
pip install -r backend/requirements.txt
```

Run backend tests:

```bash
python -m unittest backend.tests.test_rules backend.tests.test_store
```

Run the FastAPI dev server:

```bash
python -m uvicorn backend.app.main:app --host 127.0.0.1 --port 8000
```

`unittest` is part of the Python standard library, so no separate test dependency is required beyond `backend/requirements.txt`.

SQLite uses the Python standard `sqlite3` module, so no extra database dependency is required. The backend creates `backend/data/whisper_caravan.db` automatically on first startup unless `WHISPER_CARAVAN_DB_PATH` is set.

## Manual Smoke Test

### Start FastAPI

If dependencies are not installed yet:

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install -r backend/requirements.txt
```

Start the backend:

```bash
python -m uvicorn backend.app.main:app --host 127.0.0.1 --port 8000
```

### Start Next.js

In a second terminal:

```bash
npm run dev
```

### Verify `/health`

In a third terminal:

```bash
curl -s http://127.0.0.1:8000/health
```

Expected result:

```json
{"status":"ok","store":"sqlite","dbPath":".../backend/data/whisper_caravan.db"}
```

### Confirm Backend Evidence Is Being Used

1. Open the game in the browser with the FastAPI server running.
2. Make at least one choice so memories exist.
3. Check the right-side evidence panel.
4. Before Day 8, the developer badge should show:
   - `Evidence Source: Backend`
5. On Day 8, both badges should show:
   - `Evidence Source: Backend`
   - `Reaction Source: Backend`
6. Open the browser console and confirm a line like:

```text
[Whisper Caravan] retrieval sync day=8 evidence=backend reactions=backend session=...
```

### Confirm Fallback Still Works

1. Stop the FastAPI server.
2. Refresh the page or make another state change.
3. The evidence panel badges should switch to:
   - `Evidence Source: Local Fallback`
   - `Reaction Source: Local Fallback` on Day 8
4. The browser console should log a line like:

```text
[Whisper Caravan] retrieval sync day=8 evidence=local reactions=local session=...
```

### Backend Rule Test

Run the cheap backend verification suite:

```bash
python -m unittest backend.tests.test_rules backend.tests.test_store
```

This currently checks:

- inactive memories are excluded from query results
- Bear Judge rejects unsupported rumors and private inaccessible evidence
- Crow Broker can accept a public rumor
- ingested memories persist after reopening the SQLite store

## Scope

- backend ingestion and query endpoints
- metadata-aware retrieval
- structured NPC reaction responses
- evidence rendering from backend-returned data

## Recommended Backend

- FastAPI
  - memory ingestion endpoint
  - memory query endpoint
  - NPC reaction endpoint
- vector store
  - Chroma first
  - Qdrant later if needed
- structured state
  - SQLite first
  - Postgres later if needed

## Required Behavior

- Memories are sent to the backend.
- Memories are embedded and stored with metadata.
- Retrieval supports metadata filtering.
- NPC reaction endpoint returns structured JSON.
- Evidence panel displays retrieved memories with source metadata.

## Required Metadata Fields

- `memoryId: string`
- `day: number`
- `type: MemoryType`
- `source: string`
- `sourceNpcId?: string`
- `location: string`
- `faction?: string`
- `visibility: MemoryVisibility`
- `reliability: number`
- `active: boolean`
- `expiresOnDay?: number`
- `tags: string[]`

## Required NPC Reaction Response

```ts
type NPCReaction = {
  npcId: string;
  npcName: string;
  dialogue: string;
  trustDelta: number;
  priceModifier: number;
  questAvailable: boolean;
  legalRiskDelta: number;
  evidence: RetrievedEvidence[];
};
```

## Success Criteria

- Backend retrieval can reproduce the current authored consequence loop.
- Metadata filters are explicit and debuggable.
- The UI remains able to explain why evidence was returned.

## Non-Goals

- No fully dynamic story generation yet
- No complex agent orchestration yet
- No long-term player accounts
- No production deployment requirement yet
