# v0.4 Scope

## Intent

Replace deterministic mock retrieval with a real retrieval service while keeping the consequence model interpretable.

## Status

v0.4.0 slice 1 is implemented:

- FastAPI backend skeleton
- `/health`
- `/memories/ingest`
- `/memories/query`
- `/npc/reaction`
- in-memory store only
- metadata filtering only
- frontend retrieval adapter
- local deterministic fallback preserved
- backend-returned evidence metadata shown in UI

v0.4.0 slice 2 is implemented:

- SQLite persistence for ingested memory metadata
- same ingest/query/reaction API contract preserved
- deterministic retrieval rules remain in `backend/app/rules.py`
- local development database path defaults to `backend/data/whisper_caravan.db`

v0.4.0 slice 3 is implemented:

- Chroma-backed vector retrieval candidate layer
- local deterministic embedding provider, no API key required
- Chroma only suggests candidate memory ids
- SQLite remains the authoritative source of truth
- deterministic metadata and NPC filters remain authoritative
- automatic fallback to SQLite-only deterministic retrieval when Chroma is unavailable
- backend query/reaction responses now include optional developer-facing retrieval debug metadata

v0.4.0 slice 4 step 1 is implemented:

- deterministic post-filter ordering for query evidence results
- deterministic ordering within accepted/rejected NPC reaction evidence lists
- final response order now depends only on authoritative evidence fields
- vector candidate order no longer determines final backend evidence order
- local fallback mirrors the same ordering policy for evidence and Day 8 reaction lists

v0.4.0 slice 4 debug/manual pass is implemented:

- developer-facing evidence/reaction debug labels now separate response source from candidate mode more clearly
- browser console retrieval logs now report `evidenceSource` / `evidenceMode` and `reactionSource` / `reactionMode` as separate fields
- manual Day 8 smoke-test guidance now explicitly covers backend/vector, backend/sqlite fallback, local fallback, and deterministic ordering checks
- a small retrieval test now locks in the current candidate-span behavior when no explicit query limit is provided
- candidate/index text tuning and query text tuning were inspected and intentionally deferred because current vector requests still ask for the full session memory set when no explicit limit is provided, so text-only tuning would not materially change final playtest evidence membership or ordering

v0.4.0 slice 4 candidate breadth pass is implemented:

- backend vector retrieval now caps default candidate breadth at `8` when request `limit` is absent
- explicit request `limit` still uses the widened candidate policy `max(limit * 3, limit, 8)` up to the session memory count
- candidate/index text now encodes deterministic authoritative fields for retrieval, including day, source, location, visibility, reliability, evidence role, active status, and tags
- backend query/reaction text now encodes deterministic request and NPC context so narrowed candidate retrieval has meaningful matching signals
- SQLite still resolves authoritative rows before deterministic rules apply
- deterministic post-filter evidence ordering still controls final evidence order after narrowed vector retrieval

Not started yet in v0.4:

- semantic retrieval
- LLM generation
- broader retrieval-quality tuning beyond the current default candidate cap and deterministic text helpers

## Backend Setup

Create a local virtual environment:

```bash
python3 -m venv .venv
```

Activate it:

```bash
source .venv/bin/activate
```

Install backend dependencies:

```bash
pip install -r backend/requirements.txt
```

Run backend tests:

```bash
python -m unittest backend.tests.test_rules backend.tests.test_store backend.tests.test_retrieval
```

Run the FastAPI dev server:

```bash
python -m uvicorn backend.app.main:app --host 127.0.0.1 --port 8000
```

`unittest` is part of the Python standard library, so no separate test dependency is required beyond `backend/requirements.txt`.

SQLite uses the Python standard `sqlite3` module, so no extra database dependency is required. The backend creates `backend/data/whisper_caravan.db` automatically on first startup unless `WHISPER_CARAVAN_DB_PATH` is set.

Chroma persists local vector data under `backend/data/chroma` by default unless `WHISPER_CARAVAN_CHROMA_PATH` is set.

This slice uses a local deterministic embedding provider behind `backend/app/embeddings.py`, so no external embedding API key is required.

8. In the evidence panel, confirm the developer badges show:

   - `candidate mode vector`

9. On Day 8, confirm the reaction badge also shows:
   - `Reaction Source: Backend`

## Slice 4 Step 1 Ordering Policy

This step changes ordering only. It does not change which evidence survives filtering.

Final evidence ordering now uses authoritative fields in this order:

1. newer `day` first
2. higher `reliability` first
3. `title` ascending as a stable text tie-breaker
4. `source` ascending as a stable text tie-breaker
5. `memoryId` ascending as the final stable tie-breaker

For NPC reactions, accepted/rejected grouping stays unchanged. The same ordering policy is applied only within each list.

## Slice 4 Debug/Manual Notes

Earlier debug-pass finding:

- full-session candidate breadth made vector text tuning largely irrelevant to the normal playtest path

Current accepted behavior:

- backend query and reaction vector calls now cap candidates to `8` when request `limit` is absent
- explicit request `limit` still uses `max(limit * 3, limit, 8)` up to the session memory count
- because candidate breadth is now narrower in larger runs, deterministic candidate/index text and query text now influence which ids enter SQLite resolution

## Authority Model

Slice 3 does not make Chroma authoritative.

Query / reaction flow:

1. Receive query or NPC request.
2. Use Chroma only to fetch candidate memory ids.
3. Resolve candidate ids through SQLite.
4. Apply deterministic filters from `backend/app/rules.py`.
5. Return the same structured evidence / reaction shape as earlier slices.

If Chroma is unavailable, the backend falls back to SQLite-only deterministic retrieval.

## Scope

- backend ingestion and query endpoints
- metadata-aware retrieval
- structured NPC reaction responses
- evidence rendering from backend-returned data

## Recommended Backend

- FastAPI
  - memory ingestion endpoint
  - memory query endpoint
  - NPC reaction endpoint
- vector store
  - Chroma first
  - Qdrant later if needed
- structured state
  - SQLite first
  - Postgres later if needed

## Required Behavior

- Memories are sent to the backend.
- Memories are embedded and stored with metadata.
- Retrieval supports metadata filtering.
- NPC reaction endpoint returns structured JSON.
- Evidence panel displays retrieved memories with source metadata.

## Required Metadata Fields

- `memoryId: string`
- `day: number`
- `type: MemoryType`
- `source: string`
- `sourceNpcId?: string`
- `location: string`
- `faction?: string`
- `visibility: MemoryVisibility`
- `reliability: number`
- `active: boolean`
- `expiresOnDay?: number`
- `tags: string[]`

## Required NPC Reaction Response

```ts
type NPCReaction = {
  npcId: string;
  npcName: string;
  dialogue: string;
  trustDelta: number;
  priceModifier: number;
  questAvailable: boolean;
  legalRiskDelta: number;
  evidence: RetrievedEvidence[];
};
```

## Success Criteria

- Backend retrieval can reproduce the current authored consequence loop.
- Metadata filters are explicit and debuggable.
- The UI remains able to explain why evidence was returned.

## Non-Goals

- No fully dynamic story generation yet
- No complex agent orchestration yet
- No long-term player accounts
- No production deployment requirement yet
